PROBLEM CONTEXT:
- User reported critical issue: "Operations in service not finding vm created in api.v1.vms"

- ServiceController was only using WMI backend, ignoring HCS VMs
- This broke the unified VM management functionality

ROOT CAUSE ANALYSIS:
- ServiceController had both HCS and WMI services injected

- HCS VMs created via /api/v1/vms were stored in HCS VmService but invisible to service operations

SOLUTION IMPLEMENTED:
✅ Updated ServiceController to support unified VM management
✅ Modified all VM operation methods to try HCS first, then fall back to WMI
✅ Maintained backward compatibility with existing WMI VMs
✅ Added proper backend identification in responses

DETAILED CHANGES:

1. ServiceController.cs - Updated VM Operation Methods:
   - StartVm: Now checks HCS first, falls back to WMI
   - StopVm: Now checks HCS first, falls back to WMI  
   - TerminateVm: Now checks HCS first, falls back to WMI
   - PauseVm: Now checks HCS first, falls back to WMI

2. Implementation Pattern:
   ```csharp
   // Try HCS first
   if (_hcs.IsVmPresent(name))

       _hcs.OperationMethod(name);
       return Ok();
   }
   
   // Fall back to WMI
   if (_repl.IsVmPresent(name))
   {
       _repl.OperationMethod(name);
       return Ok();
   }
   
   return NotFound();
   ```

3. Special Handling:

   - SaveVm: Returns 501 for HCS VMs (container-like behavior doesn't support save state)
